## 🟣 Agent範式

⚠️ **本期最強訊號集中在此櫃**。三張熱點卡，共同指向一件事：**agent 的競爭焦點正從「怎麼搭架構」轉到「怎麼讓 agent 持續變強、且在 production 可控」**。

### 🔥 熱點卡 1 · Agent Memory ＝ 持續學習 ＝「Sleep-time compute / 做夢」
🔥 ~8 人在談（Harrison Chase 主導整週推播）· ⚠️ 範式訊號

> **一句話**：業界正把「agent memory」重新定義為「持續學習」——讓 agent 在每次跑完後，用一段「離線運算（sleep-time compute / dreaming）」回看自己的 trace、萃取教訓、更新記憶，下次更強。這正是 LangSmith 推出 Engine 的賣點。

- 🏢 **直接影響 ctkpro 的 agent 工作流設計**：若接案要交付「會隨用越用越準」的 agent（客服／內部知識助手），這條範式給了具體配方——agent 跑 → 分析 trace → 更新 memory 的三步迴圈。可評估是否值得在交付物裡內建這層，當差異化賣點。
- 🧠 **個人能力前緣**：這是「人＋agent 協作」從「每次重來」走向「累積式協作」的關鍵一躍，值得我親手試一輪（拿自己的 Claude 工作流配個 memory 迴圈）。
- 多方觀點：@hwchase17 主推「LangSmith Engine as Sleep Time Compute」、拆解 memory 三步迴圈（run→analyze traces→update memory）｜ @sarahwooders（Letta）「sleep-time compute 是 intelligence 的下一個 scaling 軸」、主張此概念源自 Letta｜ @jakebroekhuizen「架構基本已解決，未解的是 memory」｜ Sydney Runkle「memory 是 agent 持續學習的關鍵」
- 🔗 [LangSmith Engine](https://x.com/hwchase17/status/2069857129272627626)｜[memory 三步迴圈](https://x.com/hwchase17/status/2069856656335556766)｜[Letta sleep-time compute](https://x.com/hwchase17/status/2069866841615536153)｜[Jake：未解的是 memory](https://x.com/hwchase17/status/2069876360261386320)

### 🔥 熱點卡 2 · Self-Harness：讓 agent 自己改造自己的工具殼（loop engineering）
🔥 ~5 人在談 · ⚠️ 範式訊號

> **一句話**：新 paper「Self-Harness — Harnesses that improve themselves」——agent 不只執行任務，還能觀察自己哪裡弱（weakness mining）、自動修改自己的 harness（提示、工具、流程）來補強。核心是「loop engineering」：用 Data/Evals/Feedback 給出梯度，讓 agent 沿著它「自我爬坡」。

- 🏢 **影響交付的 agent 維運模式**：傳統交付完 agent 就靠人工調 prompt／加工具；self-harness 指向「agent 自帶改進迴圈」，長期可降低維運人力——但 Harrison 同步警告「loop 會放大行為，是雙面刃」，沒有好的 eval/feedback 梯度反而會放大錯誤。對 ctkpro＝先把 eval 基建做好，才談自動自我改進。
- 🧠 與我的 OODA／迴圈思維直接共鳴：weakness mining→update harness→re-run 本質是 agent 版的 OODA（觀察→定向→決策→行動）。
- 多方觀點：@hwchase17 轉介此 paper（非 LangChain 出品但讚賞）｜ Sydney Runkle 拆解步驟「weakness mining → 改 harness」｜ Kangwook Lee「讓聰明的馬自己造馬具」｜ Viv 提醒「auto-research 迴圈只有在 Data/Evals/Feedback 給出有用梯度時才有效」
- 🔗 [Self-Harness paper 介紹](https://x.com/hwchase17/status/2069443268593537470)｜[Sydney 拆步驟](https://x.com/hwchase17/status/2069489233182495071)｜[Vir：需要梯度才有效](https://x.com/hwchase17/status/2069471929942020531)

### 🔥 熱點卡 3 · Production Agent 的「治理／安全邊界」討論成形
🔥 ~6 則撞題（HN「AI agent」雷達）· ⚠️ 範式訊號（偏防守面）

> **一句話**：當 agent 真的跑進 production，社群焦點從「能不能做」轉到「做錯了怎麼辦／誰負責／哪一步該人類批准」——出現 Agent Management Platform（AMP）這個新品類，討論「approval prompt 何時算 security boundary」「agent 風險在 composition graph 而非 SBOM」「human override 該怎麼記錄」。

- 🏢 **直接打到 ctkpro 接案的責任邊界**：若交付能自主執行的 agent 給客戶，「哪些動作要人類點頭、出事怎麼追、override 如何留痕」會變成合約／SOP 的必答題。這是把 agent 賣進企業的門檻，也是可收費的差異化（治理層）。提早建立一套「agent 動作分級＋審批＋稽核」SOP，就是護城河。
- 🧠 觀察 production agent 從「炫技 demo」進入「可問責工程」階段——這是技術成熟度訊號。
- 多方觀點：execlave 連發「什麼是 AMP」「治理 vs 可觀測性」（自家宣傳，但品類訊號成立）｜「approval prompt 何時是 security boundary」gist 討論｜「你的 agent 風險在 composition graph，不是 SBOM」｜「該怎麼記錄人類對 agent 輸出的 override」｜反面：自主 agent 在 Hoppscotch 找到 CVSS 10.0 全面入侵漏洞（agent 既是風險也是稽核工具）
- 🔗 [approval prompt 是否為安全邊界](https://news.ycombinator.com/item?id=48651375)｜[composition graph not SBOM](https://openaca.dev/blog/your-agent-risk-is-in-the-composition/)｜[human override 怎麼記錄](https://news.ycombinator.com/item?id=48658038)｜[autonomous agent 找到 CVSS 10.0](https://github.com/hoppscotch/hoppscotch/security/advisories/GHSA-j542-4rch-8hwf)
